# 1. Problem Setting

## 1-1. Problem definition

EvoArenaк°Җ лӢӨлЈЁлҠ” н•өмӢ¬ settingмқҖ persistent environment evolutionмқҙлӢӨ.

Environment versionмқ„ лӢӨмқҢмІҳлҹј лӮҳнғҖлӮј мҲҳ мһҲлӢӨ.

$$
\mathcal{E}^{(1)}, \mathcal{E}^{(2)}, \ldots, \mathcal{E}^{(T)}
$$

к°Ғ versionмқҖ лҸ…лҰҪм Ғмқё random taskк°Җ м•„лӢҲлӢӨ. к°ҷмқҖ high-level settingмқ„ кіөмң н•ҳл©ҙм„ң мқҙм „ changeлҘј мқҙм–ҙл°ӣлҠ”лӢӨ.

$$
\mathcal{E}^{(t)}
=
\mathrm{Update}
\left(
\mathcal{E}^{(t-1)},
\delta^{(t)}
\right)
$$

м—¬кё°м„ң $\delta^{(t)}$лҠ” path, interface, dependency, code, validation rule, preferenceмІҳлҹј нҳ„мһ¬ versionм—җм„ң мғҲлЎң м Ғмҡ©лҗң changeлӢӨ.

AgentлҠ” version $t$мқҳ taskлҘј н’Җ л•Ң м„ё к°Җм§ҖлҘј кө¬л¶„н•ҙм•ј н•ңлӢӨ.

1. мғҲлЎң л°”лҖҗ кІғмқҖ л¬ҙм—Үмқёк°Җ.
2. мқҙм „ versionм—җм„ң м—¬м „нһҲ мң нҡЁн•ң кІғмқҖ л¬ҙм—Үмқёк°Җ.
3. мқҙм „ behavior мӨ‘ лҚ” мқҙмғҒ м Ғмҡ©н•ҳл©ҙ м•Ҳ лҗҳлҠ” кІғмқҖ л¬ҙм—Үмқёк°Җ.

мқҙ л¬ём ңлҠ” лӢЁмҲң continual learningкіјлҸ„ мЎ°кёҲ лӢӨлҘҙлӢӨ. Agent weightлҘј кі„мҶҚ fine-tuningн•ҳлҠ” кІғмқҙ м•„лӢҲлқј, к°ҷмқҖ agentк°Җ interaction historyмҷҖ memoryлҘј мӮ¬мҡ©н•ҙ current versionм—җ л§һкІҢ н–үлҸҷн•ҙм•ј н•ңлӢӨ.

лҳҗн•ң environmentлҠ” episode лҸ„мӨ‘ л¬ҙмһ‘мң„лЎң ліҖн•ҳм§Җ м•ҠлҠ”лӢӨ. EvoArenaмқҳ terminalкіј software taskлҠ” versionлі„лЎң fixed episodeлҘј м ңкіөн•ңлӢӨ. Dynamicн•ҳлӢӨлҠ” лң»мқҖ **chronologically linked discrete releases**к°Җ мЎҙмһ¬н•ңлӢӨлҠ” лң»мқҙлӢӨ.

## 1-2. Static evaluation is insufficient

Static benchmarkм—җм„ңлҠ” agentк°Җ н•ҳлӮҳмқҳ snapshotм—җ мөңм Ғнҷ”лҗҳм–ҙлҸ„ лҶ’мқҖ scoreлҘј м–»мқ„ мҲҳ мһҲлӢӨ.

мҳҲлҘј л“Өм–ҙ terminal agentк°Җ лӢӨмқҢ strategyлҘј кё°м–өн•ңлӢӨкі  н•ҳмһҗ.

```text
Build the artifact in /workspace/output and deploy from main.
```

лӢӨмқҢ releaseм—җм„ң output pathк°Җ `/workspace/dist`лЎң л°”лҖҢкі  protected branch policy л•Ңл¬ём—җ `release` branchлҘј мӮ¬мҡ©н•ҙм•ј н•  мҲҳ мһҲлӢӨ. Latest memoryл§Ң лӢӨмқҢмІҳлҹј л°”кҫёл©ҙ current taskлҠ” н’Җ мҲҳ мһҲлӢӨ.

```text
Build the artifact in /workspace/dist and deploy from release.
```

н•ҳм§Җл§Ң мқҙ updateлҠ” лӢӨмқҢ м •ліҙлҘј м§ҖмҡҙлӢӨ.

- мқҙм „ versionм—җм„ңлҠ” `/workspace/output`мқҙ л§һм•ҳлӢӨ.
- Branch changeлҠ” policy update л•Ңл¬ёмқҙм—ҲлӢӨ.
- лӢӨлҘё organizationм—җм„ңлҠ” м—¬м „нһҲ `main`мқ„ мӮ¬мҡ©н•  мҲҳ мһҲлӢӨ.
- Rollbackмқҙ л°ңмғқн•ҳл©ҙ old strategyк°Җ лӢӨмӢң н•„мҡ”н•  мҲҳ мһҲлӢӨ.

Static task н•ҳлӮҳл§Ң ліҙл©ҙ мөңмӢ  ruleл§Ң мһҲмңјл©ҙ м¶©л¶„н•ҳлӢӨ. Evolution chainм—җм„ңлҠ” transition contextк°Җ н•„мҡ”н•ҳлӢӨ.

## 1-3. State collapse in latest-only memory

мқјл°ҳм Ғмқё memory updateлҠ” лӢӨмқҢкіј к°ҷмқҙ м“ё мҲҳ мһҲлӢӨ.

$$
M_t = U(M_{t-1}, x_t)
$$

- $M_{t-1}$: мқҙм „ memory
- $x_t$: мғҲ observation or feedback
- $U$: memory update function
- $M_t$: consolidated latest memory

New informationмқҙ old informationмқ„ м•Ҳм „н•ҳкІҢ supersedeн•ңлӢӨл©ҙ мқҙ кө¬мЎ°лҠ” нҡЁмңЁм ҒмқҙлӢӨ. н•ҳм§Җл§Ң version-dependent knowledgeм—җм„ңлҠ” overwriteк°Җ lossлҘј л§Ңл“ лӢӨ.

мҳҲлҘј л“Өм–ҙ лӢӨмқҢ л‘җ statementлҠ” лӢЁмҲң conflictк°Җ м•„лӢҲлӢӨ.

- Version 1: use API v1 with field `user_id`
- Version 2: use API v2 with field `account_id`

Version 2к°Җ мөңмӢ мқҙлқјлҠ” мӮ¬мӢӨл§Ң м ҖмһҘн•ҳл©ҙ API v1 compatibility taskлҘј мІҳлҰ¬н•ҳкё° м–ҙл өлӢӨ. л‘җ stateмҷҖ transition reasonмқ„ н•Ёк»ҳ м ҖмһҘн•ҙм•ј н•ңлӢӨ.

EvoArenaк°Җ м§Җм Ғн•ҳлҠ” failureлҠ” memoryк°Җ м§§м•„м„ңк°Җ м•„лӢҲлқј, **consolidationмқҙ historyлҘј м ңкұ°н•ҳлҠ” л°©мӢқмңјлЎң м„Өкі„лҗҳм–ҙ мһҲлӢӨлҠ” м җ**мқҙлӢӨ.

## 1-4. Dynamic benchmark and persistent evolution are different

Recent dynamic benchmarkлҠ” fresh task, asynchronous event, generated variantлҘј мӮ¬мҡ©н•ңлӢӨ. мқҙлҠ” static benchmark contaminationкіј realism л¬ём ңлҘј мӨ„мқёлӢӨ.

н•ҳм§Җл§Ң лӢӨмқҢ л‘җ settingмқҖ лӢӨлҘҙлӢӨ.

### Task refresh

- мғҲ issueк°Җ л“Өм–ҙмҳЁлӢӨ.
- мғҲ web pageк°Җ л“ұмһҘн•ңлӢӨ.
- мғҲ task sampleмқ„ нҸүк°Җн•ңлӢӨ.
- мқҙм „ taskмҷҖ direct version relationмқҙ м—Ҷм–ҙлҸ„ лҗңлӢӨ.

### Persistent evolution

- к°ҷмқҖ workflowк°Җ releaseл§ҲлӢӨ л°”лҖҗлӢӨ.
- к°ҷмқҖ repository stateк°Җ milestoneмқ„ л”°лқј лҲ„м ҒлҗңлӢӨ.
- к°ҷмқҖ userмқҳ preferenceк°Җ мӢңк°„м—җ л”°лқј л°”лҖҗлӢӨ.
- кіјкұ° evidenceмҷҖ current rule мӮ¬мқҙ relationмқ„ мқҙн•ҙн•ҙм•ј н•ңлӢӨ.

EvoArenaлҠ” л‘җ лІҲм§ёлҘј нҸүк°Җ лҢҖмғҒмңјлЎң л¶„лҰ¬н•ңлӢӨ.

# 2. Core Idea

## 2-1. Main contribution

л…јл¬ёмқҳ contributionмқҖ benchmarkмҷҖ memory method л‘җ м¶•мңјлЎң лӮҳлүңлӢӨ.

### 1) EvoArena

EvoArenaлҠ” environment evolutionмқ„ м„ё domainмңјлЎң кө¬нҳ„н•ңлӢӨ.

| Subset | What evolves | Base agent | Core capability |
| --- | --- | --- | --- |
| Terminal-Bench-Evo | Path, CLI, dependency, permission, validation | Terminus 2 | Workflow version adaptation |
| SWE-Chain-Evo | Accumulated repository state and milestones | OpenHands | New change plus regression avoidance |

### 2) EvoMem

EvoMemмқҖ existing memory updater мң„м—җ patch recording and retrievalмқ„ м¶”к°Җн•ңлӢӨ.

- Latest memoryлҠ” current consolidated stateлҘј м ңкіөн•ңлӢӨ.
- Patch historyлҠ” overwritten state and update rationaleмқ„ ліҙмЎҙн•ңлӢӨ.
- Queryк°Җ temporal, conflicting, version-specificн•  л•Ң relevant patchлҘј к°Җм ёмҳЁлӢӨ.
- Base agent architectureлҘј мҷ„м „нһҲ көҗмІҙн•ҳм§Җ м•ҠлҠ”лӢӨ.

## 2-2. Design intuition

EvoMemмқҳ н•өмӢ¬ м§ҒкҙҖмқҖ gitм—җ к°Җк№қлӢӨ.

Git repositoryм—җм„ң latest fileл§Ң ліҙл©ҙ нҳ„мһ¬ codeлҠ” м•Ң мҲҳ мһҲлӢӨ. н•ҳм§Җл§Ң bugк°Җ м–ём ң л“Өм–ҙмҷ”лҠ”м§Җ, м–ҙл–Ө requirement л•Ңл¬ём—җ lineмқҙ л°”лҖҢм—ҲлҠ”м§Җ, мқҙм „ behaviorлҘј rollbackн•  мҲҳ мһҲлҠ”м§ҖлҠ” commit historyк°Җ мһҲм–ҙм•ј м•Ң мҲҳ мһҲлӢӨ.

Agent memoryлҸ„ л§Ҳм°¬к°Җм§ҖлӢӨ.

- Latest memoryлҠ” current working treeлӢӨ.
- PatchлҠ” commit diffлӢӨ.
- RationaleмқҖ commit messageлӢӨ.
- EvidenceлҠ” issue, test failure, user interaction, environment snapshotмқҙлӢӨ.
- RetrievalмқҖ relevant history searchлӢӨ.

мқҙл•Ң лӘЁл“  observationмқ„ patchлЎң м ҖмһҘн•ҳм§Җ м•ҠлҠ”лӢӨ. Purely additive informationмқҖ base memoryм—җ лӮЁкё°кі , кё°мЎҙ memoryлҘј revise, overwrite, reinterpretн•ҳлҠ” non-additive updateл§Ң patchлЎң л§Ңл“ лӢӨ.

мқҙ м„ нғқмқҖ patch historyмқҳ signal densityлҘј лҶ’мқҙл ӨлҠ” кІғмқҙлӢӨ.

## 2-3. Memory evolution as evidence

EvoMemмқҖ update м „нӣ„ memory м°ЁмқҙлҘј кі„мӮ°н•ңлӢӨ.

$$
\Delta_t
=
\mathrm{Diff}
\left(
M_{t-1},
M_t
\right)
$$

Non-additive updateк°Җ к°җм§Җлҗҳл©ҙ patchлҘј л§Ңл“ лӢӨ.

$$
p_t
=
\left(
\tau_t,
C_t^{-},
C_t^{+},
r_t,
z_t,
e_t
\right)
$$

к°Ғ fieldлҠ” лӢӨмқҢ мқҳлҜёлҘј к°Җм§„лӢӨ.

| Field | Meaning |
| --- | --- |
| $\tau_t$ | Turn, session, timestamp к°ҷмқҖ temporal metadata |
| $C_t^{-}$ | Update м „ affected memory content |
| $C_t^{+}$ | Update нӣ„ affected memory content |
| $r_t$ | мҷң updateн–ҲлҠ”м§Җм—җ лҢҖн•ң rationale |
| $z_t$ | Changeмқҳ concise semantic summary |
| $e_t$ | Triggering interaction, task feedback, snapshot к°ҷмқҖ evidence |

Patch historyлҠ” append-only collectionмңјлЎң мң м§ҖлҗңлӢӨ.

$$
\mathcal{P}_{1:t}
=
\left\{
p_1,
p_2,
\ldots,
p_t
\right\}
$$

мқҙ кө¬мЎ°м—җм„ң memoryлҠ” л‘җ мёөмңјлЎң лӮҳлүңлӢӨ.

1. Current state: $M_t$
2. Evolution trace: $\mathcal{P}_{1:t}$

Current stateл§ҢмңјлЎң м¶©л¶„н•ң queryм—җлҠ” patchлҘј мӮ¬мҡ©н•ҳм§Җ м•Ҡм•„лҸ„ лҗңлӢӨ. Historical compatibility, conflict, rollback, temporal reasoningмқҙ н•„мҡ”н•ң queryм—җм„ң patchк°Җ м¶”к°Җ evidenceк°Җ лҗңлӢӨ.

## 2-4. Patch-augmented retrieval

Inferenceм—җм„ң base memory retrievalмқҖ к·ёлҢҖлЎң мң м§Җн•ңлӢӨ.

$$
c_{\mathrm{mem}}
=
R_{\mathrm{mem}}
\left(
q,
M_T
\right)
$$

Patch retrieverлҠ” queryмҷҖ relevantн•ң update historyлҘј м°ҫлҠ”лӢӨ.

$$
\mathcal{P}_q
=
R_{\mathrm{patch}}
\left(

# 6. Limitations

1. **Evolution is discrete, not within-episode continuous**
   - к°Ғ version episodeлҠ” fixed environmentлӢӨ.
   - Productionм—җм„ң tool or policyк°Җ мӢӨн–ү мӨ‘ л°”лҖҢлҠ” asynchronous driftлҠ” лӢӨлЈЁм§Җ м•ҠлҠ”лӢӨ.

2. **Terminal evolution is partly synthetic**
   - UpdateлҠ” realistic taxonomy and oracle validationмқ„ мӮ¬мҡ©н•ҳм§Җл§Ң naturally occurring release logл§ҢмңјлЎң кө¬м„ұлҗң кІғмқҖ м•„лӢҲлӢӨ.
   - Agent-assisted construction biasк°Җ мһҲмқ„ мҲҳ мһҲлӢӨ.

3. **SWE next state uses reference patch**
   - Agentмқҳ мӢӨм ң prior patchк°Җ next repository stateк°Җ лҗҳм§Җ м•ҠлҠ”лӢӨ.
   - Real deploymentмқҳ error compounding and repair debtлҘј м ңкұ°н•ң settingмқҙлӢӨ.

4. **SWE chain metric definition is unusual**
   - Consecutive solved prefix ratioлҘј chain accuracyлқјкі  л¶ҖлҘёлӢӨ.
   - Terminal and Personaмқҳ all-or-nothing chain metricкіј м§Ғм ‘ л№„көҗн•ҳкё° м–ҙл өлӢӨ.

5. **Persona data is synthetic and small in persona count**
   - 10 persona conversationм—җ 505 questionмқ„ кө¬м„ұн•ңлӢӨ.
   - Real user preference change, ambiguity, consentлҘј м¶©л¶„нһҲ лҢҖн‘ңн•ҳм§Җ лӘ»н•  мҲҳ мһҲлӢӨ.

6. **MCQ format limits personalization realism**
   - Real assistantлҠ” free-form recommendation and actionмқ„ н•ҙм•ј н•ңлӢӨ.
   - Balanced option accuracyк°Җ user satisfactionмқ„ м§Ғм ‘ лҢҖліҖн•ҳм§Җ м•ҠлҠ”лӢӨ.

7. **EvoMem gains are modest and non-uniform**
   - EvoArena average step gainмқҖ м•Ҫ +1.5 pointлӢӨ.
   - мқјл¶Җ backbone and question typeм—җм„ңлҠ” regressionмқҙ л°ңмғқн•ңлӢӨ.

8. **Mechanism analysis is observational**
   - Patch uptakeмҷҖ success correlationмқҙ causal interventionмңјлЎң мҷ„м „нһҲ кІҖмҰқлҗң кІғмқҖ м•„лӢҲлӢӨ.

9. **Patch trigger quality is a new bottleneck**
   - Non-additive updateлҘј мһҳлӘ» к°җм§Җн•ҳл©ҙ important historyлҘј лҶ“м№ҳкұ°лӮҳ unnecessary patchлҘј л§Ңл“ лӢӨ.

10. **Patch retrieval can revive stale behavior**
    - Similarityк°Җ лҶ’лӢӨлҠ” мқҙмң л§ҢмңјлЎң old ruleмқ„ к°Җм ёмҳӨл©ҙ current versionм—җ л§һм§Җ м•Ҡмқ„ мҲҳ мһҲлӢӨ.
    - Version scope and validity metadataк°Җ н•„мҡ”н•ҳлӢӨ.

11. **Append-only history grows indefinitely**
    - Storage, indexing, context cost, duplicate patch л¬ём ңк°Җ мғқкёҙлӢӨ.
    - Compaction policyк°Җ н•„мҡ”н•ҳлӢӨ.

    - Userк°Җ old preference мӮӯм ңлҘј мҡ”мІӯн•ҙлҸ„ patch historyм—җ лӮЁмқ„ мҲҳ мһҲлӢӨ.
    - Right-to-delete and audit requirement мӮ¬мқҙ м •мұ…мқҙ н•„мҡ”н•ҳлӢӨ.

13. **Persistent poisoning risk**
    - Prompt injection or malicious tool feedbackк°Җ patchлЎң м ҖмһҘлҗҳл©ҙ future sessionк№Ңм§Җ мҳҒн–Ҙмқ„ мӨ„ мҲҳ мһҲлӢӨ.

    - Model-generated reasonмқҙ actual causal explanationкіј лӢӨлҘј мҲҳ мһҲлӢӨ.
    - Trigger evidenceмҷҖ explanationмқ„ л¶„лҰ¬н•ҙм„ң м ҖмһҘн•ҙм•ј н•ңлӢӨ.

    - Cross-domain averageлҠ” method generalityлҘј ліҙм—¬мЈјм§Җл§Ң exact comparabilityлҠ” м ңн•ңм ҒмқҙлӢӨ.

    - мқјл¶Җ subset deltaлҸ„ prose and rounded tableмқҙ лӢӨлҘҙкІҢ ліҙмқёлӢӨ.

    - Repository top-level READMEлҠ” мқјл¶Җ experiment folderк°Җ мҲңм°Ё кіөк°ңлҗңлӢӨкі  лӘ…мӢңн•ңлӢӨ.
